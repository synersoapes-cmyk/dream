# 梦幻实验室项目说明

## 1. 项目定位

梦幻实验室是一个面向《梦幻西游》角色养成与换装决策的数值分析系统。

系统目标不是单纯记录角色面板，而是围绕以下三个核心场景展开：

- 同步玩家当前角色状态
- 识别并管理候选装备
- 在实验室中推演换装后的实际收益

最终输出包括：

- 面板属性变化
- 技能等级变化
- 单次或单秒伤害变化
- 总伤增益百分比
- 性价比判断（含跨服费、1 点法伤成本、低性比提醒）

文档分工如下：

- 当前系统说明见 [project-overview.md](/Users/czy/Documents/dream/docs/project-overview.md)
- 实现状态、缺口和开发任务见 [implementation-task-list.md](/Users/czy/Documents/dream/docs/implementation-task-list.md)
- 测试验收、专项回归和发布链路见 [simulator-test-report.md](/Users/czy/Documents/dream/docs/simulator-test-report.md)

## 2. 核心页面结构

文档中定义了两大主模块：

- 当前状态
- 实验室

### 2.1 当前状态

目标：构建角色的基准档案，同步游戏现状。

包含三个子模块：

- 个人档案
- 当前装备
- 战斗参数

当前实现补充：

- 登录后会读取 `/api/simulator/current`；若当前用户还没有 simulator 角色，服务端会按默认模板自动补建初始角色与默认战斗参数。
- 当前状态页的资料、修炼、装备、战斗参数都已经接入 D1 真实读写，不再只依赖本地 store。
- 技能伤害面板会联动当前目标模板 / 副本目标参数，当前状态页与实验室不会再各自维护一套脱节的目标值。

### 2.2 实验室

目标：做换装推演与购买决策。

包含三个子模块：

- 装备库
- 实验对比中心
- 预期效果预判

当前实现里，实验室左侧的“待确认新品 / 新品装备库”已经接入 D1 持久化，使用独立的候选装备表保存候选库存。
当前实现里，右侧 AI 顾问面板已经接入 Gemini，会结合当前角色、候选装备和实验室席位上下文回答配装问题。
当前实现里，实验室支持把单件或整套实验结果直接应用到当前装备；应用前会自动生成 `应用前快照`，并允许回滚最近一次应用。
当前实现里，OCR 已接入 `Gemini + R2`，后台已提供 OCR 健康检查、OCR 作业、OCR 字典、候选装备库与入库台账页面。
当前实现里，前台 OCR 已统一先写入 `ocr_job -> ocr_draft_item` 作为审计链，再自动同步到 `candidate_equipment.pending`；当前不再依赖后台逐条人工处理才能进入候选装备库。
当前实现里，后台治理页已继续补齐到第二阶段：新增了装备扩展规则工作台、OCR 质量统计页和顾问问答审计页。
当前后台已覆盖导航、规则中心、星相互合规则、默认模板、目标模板、OCR 运营页、候选装备库、入库台账、实验室记录、用户排障与顾问配置。
当前实验室正式口径为“样本席位 + 2 个对比席位”，并已补齐继承模拟、伤害差额显示、横向对比和替换确认主链。
详细验收证据见 [simulator-test-report.md](/Users/czy/Documents/dream/docs/simulator-test-report.md)。

### 2.3 当前前台落地差距补充

相对最新产品说明，当前前台还存在以下关键差距：

- 当前装备与实验室详情已补出独立“初值”展示区，但更强的规则解释和套装成因提示仍需继续补齐。
- 当前装备页与实验室席位已补出“双加汇总”摘要，未开孔符石组合也会显式显示 `未激活`；剩余缺口主要是更完整的套装阶梯解释，而不是基础汇总缺失。
- `M4-08` 当前先采用一版临时口径：按“动物套 / 变身套”的同名穿戴件数计算，`3 件 = 魔力 +10`、`5 件 = 魔力 +20`，并要求同时进入当前装备面板、实验室对比和服务端伤害试算；若后续产品给出正式数值，再整体替换。
- OCR 主链虽然已经接入确认弹窗与彩色 diff，但针对“藏宝阁截图 / 聊天框预览图”的专门识别模式和前台反馈仍需继续细化。
- 实验室当前正式口径已经统一为“样本席位 + 2 个对比席位”，并支持“继承旧宝石 / 继承旧符石”切换；剩余缺口主要在更严格的装备规则校验，而不是基础交互缺失。
- 实验室差值表、涨跌差值和“覆盖到当前装备”闭环已经真实可用；席位卡底部现已补出固定口径的“`伤害提升：xx点 / 伤害下降：xx点`”结果项。
- 当前已支持 `2` 个对比席位横向优选，实验室主界面、会话恢复与顾问上下文都会同步保留这两个席位。
- 性价比模型已经统一为“服务端总伤 / 总价格（含跨服费）/ 1 点法伤成本”，当前也已补出“总伤增益百分比”“收益摘要”和低性比提醒。
- 主流副本目标模板仍需继续补齐覆盖率。
- 战斗环境与特殊目标规则已补入首版正式链路：当前支持 `通天河` fallback 模板、目标法防结果、雨天天气修正、目标防御状态记录，以及特殊目标法术减伤系数；这些参数已经可以在“战斗参数”页和“技能伤害”弹窗里直接调节。
- 当前状态页已补齐“快速加点”“强身”“经脉补偿”三条主链；当前这组剩余工作主要不是基础联动缺失，而是后续若要继续增强时补成更完整的经脉树编辑与说明能力。
- 其余阵法的自身伤害倍率、速度副作用和站位差异仍待补完正式规则。

- 1 份角色基础档案

- 2 件玉魄
- 技能等级
- 修炼等级
- 战斗环境参数

### 3.1 角色基础档案

基础档案至少包含：

- 门派
- 等级
- 五围加点

  - 魔力
  - 力量
  - 耐力
  - 敏捷
- 潜力点
- 核心面板属性
  - 命中
  - 伤害
  - 法伤
  - 法防
  - 速度
  - 气血
  - 魔法

### 3.2 技能与修炼

系统需要记录：

- 师门技能名称与等级
- 修炼等级
  - 攻修
  - 法修
  - 防修
  - 法抗等

其中技能等级不是静态值，必须支持符石组合带来的额外等级加成，例如：

- `180 + 2`

## 4. 采集方式

文档要求支持通过截图进行识别：

- 全屏截图
- 局部截图
- 藏宝阁装备图

OCR 识别后的数据不会直接进入正式库，而是先进入待确认区和审计链，再由用户确认是否入库。

## 7. 实验室模块

实验室是项目的核心价值区。

### 7.1 装备库

装备库分为：

- 待确认区
- 新品库

其中：

- 待确认区：存放刚识别但未确认入库的新品
- 新品库：已分类存放的候选装备，可带价格信息

当前实现补充：

- `ocr_draft_item` 负责承接 OCR 结构化草稿与审计记录
- `candidate_equipment` 继续作为实验室可直接消费的候选装备库
- 前台 OCR 成功后会自动把草稿标记为 `approved`，并同步生成候选装备记录
- `inventory_equipment_asset` 已作为正式资产层落地，承接确认入库后的装备快照、名称、部位和来源信息
- `inventory_entry` 继续作为正式库存台账，并在候选装备确认时自动指向对应正式资产
- 后台 `入库台账` 页面已支持修正价格、分类键和台账状态（已入库 / 已售出 / 已作废）
- 当后台把台账状态改为 `已售出 / 已作废` 时，会通过正式资产反查来源候选，并同步标记候选为 `replaced`，避免继续出现在实验室新品库中

### 7.2 实验对比中心

实验对比中心包含两类席位：

- 样本席位
- 对比席位

规则如下：

- 样本席位默认锁定为当前状态的全身数据
- 当前正式实现为 `2` 个对比席位，支持做单变量测试与横向优选
- 支持勾选是否继承老装备的符石/宝石
- 实验室会话本身需要独立持久化，不能只存在前端内存

### 7.3 预期效果预判

输出包括：

- 单秒伤害提升
- 总伤增益百分比
- 性价比模型

性价比目标是自动计算：

- 提升 1 点伤害需要多少梦幻币或人民币

### 7.4 同步决策

如果用户确认替换：

- 实验室方案将覆盖当前状态页

## 8.3 招云套装

### 触发条件

当全身 `6 件装备` 的第一颗符石颜色符合固定序列，且各部位满足对应星相条件时触发。

组合颜色序列为：

- 红
- 金
- 黑
- 蓝
- 红

按部位对应如下：

- 头盔：白色
- 项链：红色
- 武器：金色（黄色）
- 衣服：黑色
- 腰带：蓝色
- 鞋子：红色

### 套装效果

- 额外灵力 `+6`
- 使用 `龙卷雨击` 时，对每个目标额外增加：
  - `[目标速度 × 4%]`

文档中“灵力”的后台映射口径是：

- `+6 灵力`
- 等价于 `+6 法伤`
- 等价于 `+6 法防`

## 8.5 星相互合

星相互合规则分两层：

- 单件装备满足时，获得该部位加成
- `6 件装备全部满足` 时，触发全局加成

全局加成写死为：

- 全基础属性 `+2`

具体包含：

- 力量 `+2`
- 敏捷 `+2`
- 体质 `+2`
- 耐力 `+2`
- 魔力 `+2`

### 已明确的部位示例

头盔：

- 对应组合：`九龙诀`
- 需求颜色：`白 + 红 + 黄 + 蓝 + 绿`
- 达成后加成：`法术伤害 +2`

项链：

- 对应组合：`龙腾`
- 需求颜色：`黑 + 红 + 白 + 蓝 + 紫`
- 达成后加成：`法术伤害 +2`

## 8.6 玉魄计算规则

玉魄支持固定值和百分比两种属性。

文档中给出的明确案例：

- 如果在实验室挂载了 `法术忽视 5%` 的玉魄
- 公式修正为：
  - `(面板法伤 - 目标法防 * 0.95)`
- 如果挂载了 `基础法术伤害 +1.5%`
- 公式修正为：
  - `(面板法伤 * 1.015 - 目标法防)`

这意味着：

- 玉魄的百分比属性不能在入库时被拍平
- 必须在实验室计算阶段参与公式

## 9. 灵饰结构

灵饰分为前排与后排两类。

### 9.1 前排灵饰

- 戒指
- 耳饰

主属性：

- 戒指：物理伤害 或 物理防御

- 物理伤害
- 法术伤害
- 固定伤害
- 速度
- 物理暴击等级
- 法术暴击等级
- 法术伤害结果
- 治疗效果
- 封印命中等级
- 狂暴等级
- 穿刺等级

### 9.2 后排灵饰

- 手镯
- 佩饰

主属性：

- 佩饰：速度
- 手镯：封印命中等级 或 抵抗封印等级

附加属性可包括：

- 气血
- 防御
- 法术防御
- 格挡值
- 气血回复效果
- 抵抗封印等级
- 抗物理暴击等级
- 抗法术暴击等级

### 9.3 灵饰套装

当 `4 件灵饰` 点化相同特效，且总等级达到 `8 级以上` 时激活。

当前实现补充：

- 当前 `ornament_set_effect` 只记录真正达到 `4 件 + 总等级 >= 8` 的套装。

文档中明确列出的档位有：

- `8 级`
- `16 级`
- `24 级`
- `28 级`
- `32 级`

## 10. 玉魄结构

玉魄至少包括：

- 玉魄名称
- 玉魄品质
- 适用部位
- 契合等级

契合等级范围：

- `1-10 级`

核心属性包括：

- 法术伤害
  - 固定值
  - 百分比
- 法术暴击等级
- 法术忽视 / 穿透
  - 百分比
- 法术伤害结果

当前实现补充：

- 玉魄固定值属性可直接作为结构化数值录入当前状态页与实验室编辑器。
- 玉魄百分比属性现在也有结构化编辑入口，当前优先支持：
  - `法术忽视 / 穿透 %`
  - `基础法术伤害 %`
  - `魔法值上限 %`
  - `五行属性增强 %`
- 上述四类当前都已进入正式口径：
  - `法术忽视 / 穿透 %`：按目标法防忽视比例参与服务端伤害计算
  - `基础法术伤害 %`：按面板法伤百分比参与服务端伤害计算
  - `魔法值上限 %`：修正前台 / 服务端面板 `mp`
  - `五行属性增强 %`：当前按“词条元素匹配我方五行且处于克制关系时，加到五行系数”执行
- 玉魄百分比编辑器现在会优先读取规则中心 `jade_percent_semantics` 配置；无配置时才回退到内置默认选项。

文档中明确给出的示例值：

- 固定值示例：`法术伤害 +12`
- 百分比示例：`基础法术伤害 +1.5%`

## 14. 本项目的本质

这不是一个纯展示型工具，而是一个：

- 角色数字孪生系统
- 装备识别与审计系统
- 换装推演系统
- 决策辅助系统
- 后台规则配置系统

它最终服务的不是“看面板”，而是回答下面这些问题：

- 这件装备换上后能提升多少法伤
- 技能等级是否会因为符石变化发生跌落
- 总伤提升到底是多少
- 候选装备值不值得买
- 哪套方案最适合当前目标场景

## 15. 后台系统说明

数据库设计需要显式考虑后台，因为本项目大量规则不是一次性写死后就不变的。

后台的核心职责不是“管理用户”，而是“管理规则”。

### 15.1 后台要解决的问题

前台负责：

- 上传截图
- 确认识别
- 入库装备
- 做实验对比
- 查看结果

后台负责：

- 配置成长规则
- 配置技能公式
- 配置符石组合
- 配置星相互合
- 配置灵饰套装档位
- 配置玉魄属性池
- 配置怪物模板
- 配置 OCR 纠错词典
- 发布与回滚规则版本

### 15.2 为什么后台必须纳入一期设计

原因有三个：

- 数值规则本身会变化
- 以后一定会扩门派、扩技能、扩副本
- OCR 识别结果需要持续通过词典和映射修正

如果数据库一期不把后台规则中心考虑进去，后面会出现这些问题：

- 规则被写死在代码里，改一次就要发版
- 无法做规则灰度测试
- 无法回滚历史规则
- 无法审计是谁修改了公式

### 15.3 后台建议拆成三块

1. 规则中心

用于维护：

- 五围成长规则
- 套装规则
- 技能伤害公式
- 星相互合规则
- 玉魄计算规则
- 灵饰套装规则

2. 模板中心

用于维护：

- 怪物模板
- 默认角色模板
- OCR 标准词典
- 装备分类映射

3. 发布中心

用于管理：

- 草稿规则
- 已发布规则
- 回滚版本
- 修改审计日志

当前实现补充：

- 已落地的后台页包括：默认模板、规则中心、规则 playground、装备扩展规则、星相互合规则、目标模板、OCR 健康检查、OCR 作业、OCR 质量统计、OCR 字典、待确认 / 审计台、候选装备库、入库台账、顾问配置、顾问问答审计、实验室记录、用户排障。
- 后台能力目前已经覆盖“规则维护 + OCR 运营 + 用户排障 + 顾问审计”的主链；剩余缺口主要在更细的 OCR 强校验和更丰富的质量指标，而不是后台入口缺失。

### 15.4 后台配置的典型例子

例如当前文档中已经写死的：

- `基础气血来源 * 5 + 体质 * 4.5 = 面板气血`
- `1点体质 = 灵力 +0.3`
- `1点魔力 = 魔法上限 +3.5`
- `招云 = 白、红、金、黑、蓝、红`
- `腾蛟 = 白、红、红、黑、蓝、红`
- `龙卷雨击额外伤害 = 目标速度 × 4%`
- `龙腾额外伤害 = 魔法消耗 × 4%`
- `法术忽视 5% = 目标法防按 0.95 参与计算`

这些在产品第一版可以按当前值落地，但数据库和后台能力必须允许后续修改。

### 15.5 后台与前台的关系

建议采用这样的关系：

- 后台维护规则
- 前台不存公式，只读取当前生效版本
- 实验室计算时记录所使用的规则版本

这样做的好处是：

- 结果可追溯
- 历史实验可回放
- 线上问题可快速回滚

## 16. 运行时与持久化说明

### 16.1 默认档案

- 新用户首次进入 simulator 时，如果还没有角色档案，系统会自动创建一份默认角色。
- 默认角色模板优先从 D1 `config` 表读取，当前使用的键包括：
  - `simulator.default.character_meta`
  - `simulator.default.profile`
  - `simulator.default.skills`
  - `simulator.default.cultivations`
  - `simulator.default.equipments`
- 如果配置缺失，服务端会自动补入默认值；后续后台可以直接维护这些配置，而不需要修改查询逻辑。

### 16.2 云端保存

- simulator 当前已经接入 D1 真实读写链路，不再只依赖本地 mock store。
- 读取入口：
  - `GET /api/simulator/current`
- 已完成的写回入口：
  - `PATCH /api/simulator/current/profile`
  - `PATCH /api/simulator/current/cultivation`
  - `PATCH /api/simulator/current/equipment`
  - `GET /api/simulator/current/equipment/rollback`
  - `POST /api/simulator/current/equipment/rollback`
- 前端保存入口位于角色面板：
  - 属性页签可保存基础属性与战斗属性
  - 修炼页签可保存修炼等级
  - 装备面板与实验室确认替换可保存当前装备
- 战斗参数保存现在除了当前单目标上下文，还会一并写回手动目标列表、副本勾选状态和当前 tab。
- 实验室里的“单件同步到当前装备 / 整套应用到当前装备”现在会先自动创建一份 `应用前快照`，再覆写当前装备。
- 实验室头部新增“回滚最近一次应用”入口，会把最近一次自动保存的应用前快照重新写回当前 `current` 快照，而不是直接切换 `current_snapshot_id`，避免历史快照后续被继续编辑污染。
- 保存成功后，前端会使用接口返回的最新 bundle 重新灌入 store，保证页面状态与 D1 保持一致。

### 16.3 后台模板配置

- 管理后台新增了 simulator 默认模板编辑页：
  - `/admin/simulator-defaults`
- 该页面直接维护 D1 `config` 表中的 `simulator.default.*` 配置。
- 这些模板只影响后续新建档案，不会覆盖已有用户的角色数据。
- 目标模板后台页现在同时维护两类目标：
  - 副本目标模板
  - 手动目标模板
- 手动目标模板的扩展攻击/抗性字段保存在 `battle_target_template.payload_json`，首页手动目标面板会直接读取。
- 系统会预置一组副本目标模板，默认包含 `乌鸡国 / 水陆大会 / 车迟国 / 大雁塔` 四组样例副本，后续可继续在后台增删改。

## 17. 服务端伤害规则链路

当前服务端试算已经成为正式规则入口，前台技能伤害面板、实验室估值和后台规则试算共用同一套链路。

### 17.1 当前已落地能力

- 规则中心已接管 `rule_attribute`、`rule_skill_formula`、`rule_damage_modifier`、`rule_skill_bonus` 等核心规则表，服务端默认按当前生效 `rule_version` 试算。
- 当前已覆盖 `龙宫 / 法师` 的 `龙卷雨击 / 龙腾` 主链路，服务端会统一推导 `气血 / 法伤 / 法防 / 速度 / 命中 / 伤害 / 物防 / 躲避` 等正式面板。
- `POST /api/simulator/calculate-damage` 负责角色伤害试算，`POST /api/simulator/current/lab-valuation` 负责实验室总伤、差价和性价比估值。
- 技能试算与实验室估值都会继承同一份战斗上下文，包括阵法、五行、目标法防、目标速度、修炼差和附加结果项，不再各自维护一套条件。
- 服务端返回 `breakdown` 明细，前台可直接展示公式拆解与规则命中情况。

### 17.2 装备扩展规则

- 当前装备与实验室席位会把结构化宝石、符石组合、星位和星相互合配置一并持久化，服务端按这些结构化数据自动命中规则。
- 灵饰与玉魄已经独立建模，当前快照、历史回滚、默认角色补建和实验室估值都走同一套持久化链路。
- 灵饰套装规则已进入运行时，当前按“4 件同名灵饰 + 总等级档位”识别套装效果。
- 常规套装规则已进入运行时，当前优先读取 `regular_set_rules`；若后台未配置，则回退到临时口径 `3 件 +10 魔力 / 5 件 +20 魔力`。
- 玉魄百分比词条已进入正式计算，当前覆盖 `法术忽视 / 穿透 %`、`基础法术伤害 %`、`魔法值上限 %`、`五行属性增强 %`。
- 玉魄属性池和百分比语义已经接入规则中心，并开始约束前台编辑器和 OCR 确认弹窗。

### 17.3 符石、星相与战斗规则

- `招云 / 腾蛟` 两套 6 件符石套装已经接入正式伤害修正，会追加灵力收益和技能结果伤害。
- `九龙诀 / 呼风唤雨 / 破浪诀 / 逆鳞 / 龙腾` 按全身只取最高一套处理，`隔山打牛` 按最多两套生效处理。
- 当前系统已经按正式口径处理符石套装失效、额外灵力拆分、技能等级差值联动和基线收益去重。
- 星相互合与 6 件全套 +2 当前优先读取结构化配置；由于 `招云` 武器首孔颜色与 `破浪诀` 颜色清单存在规则冲突，现阶段仍保留一层折中校验口径，后续规则确认后可继续收紧。
- 实验室已支持符石技能变化、跌落预警、孔数冲突提示和整套应用前确认框。
- 实验室目标选择器与技能伤害面板都会按最终技能等级动态刷新“秒几”上限。

### 17.4 当前统一口径

- 龙宫法师属性换算已切到新口径，`baseHp`、五围转化、装备增量和阵法速度系数都会进入统一试算。
- `selfFormation / targetFormation / selfElement / targetElement` 已成为正式战斗输入源，保存时会自动推导阵法克制和五行关系。
- 当前已落地的阵法副作用中，`天覆阵` 按 `伤害 1.2 / 速度 0.9` 处理；其余阵法自身副作用仍先按 `1.0` 兜底。
- 当前 9 阵范围的克制矩阵已进入自动推导。
- 旧的面板展示字段只保留为快照展示用途，不再作为正式试算输入。
- 后台规则试算页仍允许显式传入“面板法伤覆盖值”，用于调试与规则核对，不影响正式前台试算口径。

## 18. 规则试算样例

- 后台 `Simulator` 页面新增了规则试算样例能力。
- 样例会保存规则版本、试算输入、期望结果快照和备注，持久化到 `rule_simulation_case`。
- 规则编辑后，可以直接载入样例重新试算，用同一组参数比较新旧版本结果。
- 这套机制主要用于伤害规则核对，减少分灵、五行、阵法、技能加成等配置调整带来的隐性漂移。
