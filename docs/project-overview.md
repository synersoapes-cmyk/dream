# 梦幻实验室项目说明

## 1. 项目定位

梦幻实验室是一个面向《梦幻西游》角色养成与换装决策的数值分析系统。

系统目标不是单纯记录角色面板，而是围绕以下三个核心场景展开：

- 同步玩家当前角色状态
- 识别并管理候选装备
- 在实验室中推演换装后的实际收益

最终输出包括：

- 面板属性变化
- 技能等级变化
- 单次或单秒伤害变化
- 总伤增益百分比
- 性价比判断

## 2. 核心页面结构

文档中定义了两大主模块：

- 当前状态
- 实验室

### 2.1 当前状态

目标：构建角色的基准档案，同步游戏现状。

包含三个子模块：

- 个人档案
- 当前装备
- 战斗参数

### 2.2 实验室

目标：做换装推演与购买决策。

包含三个子模块：

- 装备库
- 实验对比中心
- 预期效果预判

当前实现里，实验室左侧的“待确认新品 / 新品装备库”已经接入 D1 持久化，使用独立的候选装备表保存候选库存。
当前实现里，右侧 AI 顾问面板已经接入 Gemini，会结合当前角色、候选装备和实验室席位上下文回答配装问题。

## 3. 角色结构

系统内一名角色由以下部分组成：

- 1 份角色基础档案
- 6 件常规装备
- 4 件灵饰
- 2 件玉魄
- 技能等级
- 修炼等级
- 战斗环境参数

### 3.1 角色基础档案

基础档案至少包含：

- 门派
- 等级
- 五围加点
  - 体质
  - 魔力
  - 力量
  - 耐力
  - 敏捷
- 潜力点
- 核心面板属性
  - 命中
  - 伤害
  - 法伤
  - 法防
  - 速度
  - 气血
  - 魔法

### 3.2 技能与修炼

系统需要记录：

- 师门技能名称与等级
- 修炼等级
  - 攻修
  - 法修
  - 防修
  - 法抗等

其中技能等级不是静态值，必须支持符石组合带来的额外等级加成，例如：

- `180 + 2`

## 4. 采集方式

文档要求支持通过截图进行识别：

- 全屏截图
- 局部截图
- 藏宝阁装备图
- 聊天框预览图

OCR 识别后的数据不会直接进入正式库，而是先进入待确认区，再由用户审核确认入库。

## 5. 当前装备模块

当前装备区包含：

- 6 件常规装备
  - 头盔
  - 项链
  - 武器
  - 衣服
  - 腰带
  - 鞋子
- 4 件灵饰
- 2 件玉魄

装备详情至少展示：

- 初值
- 锻位
- 熔炼
- 开孔数
- 符石颜色
- 激活的符石组合
- 套装效果

系统还需要支持保存多套装备快照，例如：

- 任务套
- PK 套

## 6. 战斗参数模块

战斗参数用于伤害推演，需要支持：

- 我方阵法
- 五行关系
- 克制关系
- 目标怪物预设
- 目标防御
- 目标法防
- 目标气血
- 目标五行

文档明确要求：

- 内置主流副本怪物数据
- 选中目标后自动填充其面板参数

当前实现约束补充：

- 战斗参数跟随角色快照一起持久化
- 当前状态页默认维护一份 `snapshot_battle_context`
- 实验室新建会话时，以当前快照的战斗参数作为基线
- 后台默认模板现在也包含一份“默认战斗参数”，用于新用户首次补建角色时生成初始战斗环境

## 7. 实验室模块

实验室是项目的核心价值区。

### 7.1 装备库

装备库分为：

- 待确认区
- 新品库

其中：

- 待确认区：存放刚识别但未确认入库的新品
- 新品库：已分类存放的候选装备，可带价格信息

### 7.2 实验对比中心

实验对比中心包含两类席位：

- 样本席位
- 对比席位

规则如下：

- 样本席位默认锁定为当前状态的全身数据
- 对比席位支持挂载 `1-2 件` 新装备做局部变量测试
- 支持勾选是否继承老装备的符石/宝石
- 实验室会话本身需要独立持久化，不能只存在前端内存

### 7.3 预期效果预判

输出包括：

- 单秒伤害提升
- 总伤增益百分比
- 性价比模型

性价比目标是自动计算：

- 提升 1 点伤害需要多少梦幻币或人民币

### 7.4 同步决策

如果用户确认替换：

- 实验室方案将覆盖当前状态页

## 8. 已明确写死的数值规则

以下数值已经在原始文档中写明，属于当前版本的准确规则。

## 8.1 适用门派

当前版本仅针对：

- 龙宫
- 法师流派

## 8.2 五围转化规则

### 1 点体质

- 气血 `+4.5`
- 灵力 `+0.3`
- 速度 `+0.1`

其中灵力进一步映射为：

- 法术伤害 `+0.3`
- 法术防御 `+0.3`

### 1 点魔力

- 魔法上限 `+3.5`
- 灵力 `+0.7`

其中灵力进一步映射为：

- 法术伤害 `+0.7`
- 法术防御 `+0.7`

## 8.3 招云套装

### 触发条件

当全身 `6 件装备` 的第一颗符石颜色符合固定序列，且各部位满足对应星相条件时触发。

组合颜色序列为：

- 白
- 红
- 金
- 黑
- 蓝
- 红

按部位对应如下：

- 头盔：白色
- 项链：红色
- 武器：金色（黄色）
- 衣服：黑色
- 腰带：蓝色
- 鞋子：红色

### 套装效果

- 额外灵力 `+6`
- 使用 `龙卷雨击` 时，对每个目标额外增加：
  - `[目标速度 × 4%]`

文档中“灵力”的后台映射口径是：

- `+6 灵力`
- 等价于 `+6 法伤`
- 等价于 `+6 法防`

## 8.4 腾蛟套装

### 触发条件

组合颜色序列为：

- 白
- 红
- 红
- 黑
- 蓝
- 红

按部位对应如下：

- 头盔：白色
- 项链：红色
- 武器：红色
- 衣服：黑色
- 腰带：蓝色
- 鞋子：红色

### 套装效果

- 额外灵力 `+6`
- 使用 `龙腾` 时，额外增加：
  - `[魔法消耗 × 4%]`

## 8.5 星相互合

星相互合规则分两层：

- 单件装备满足时，获得该部位加成
- `6 件装备全部满足` 时，触发全局加成

全局加成写死为：

- 全基础属性 `+2`

具体包含：

- 力量 `+2`
- 敏捷 `+2`
- 体质 `+2`
- 耐力 `+2`
- 魔力 `+2`

### 已明确的部位示例

头盔：

- 对应组合：`九龙诀`
- 需求颜色：`白 + 红 + 黄 + 蓝 + 绿`
- 达成后加成：`法术伤害 +2`

项链：

- 对应组合：`龙腾`
- 需求颜色：`黑 + 红 + 白 + 蓝 + 紫`
- 达成后加成：`法术伤害 +2`

## 8.6 玉魄计算规则

玉魄支持固定值和百分比两种属性。

文档中给出的明确案例：

- 如果在实验室挂载了 `法术忽视 5%` 的玉魄
- 公式修正为：
  - `(面板法伤 - 目标法防 * 0.95)`

这意味着：

- 玉魄的百分比属性不能在入库时被拍平
- 必须在实验室计算阶段参与公式

## 9. 灵饰结构

灵饰分为前排与后排两类。

### 9.1 前排灵饰

- 戒指
- 耳饰

主属性：

- 戒指：物理伤害 或 物理防御
- 耳饰：法术伤害 或 法术防御

附加属性可包括：

- 物理伤害
- 法术伤害
- 固定伤害
- 速度
- 物理暴击等级
- 法术暴击等级
- 法术伤害结果
- 治疗效果
- 封印命中等级
- 狂暴等级
- 穿刺等级

### 9.2 后排灵饰

- 手镯
- 佩饰

主属性：

- 佩饰：速度
- 手镯：封印命中等级 或 抵抗封印等级

附加属性可包括：

- 气血
- 防御
- 法术防御
- 格挡值
- 气血回复效果
- 抵抗封印等级
- 抗物理暴击等级
- 抗法术暴击等级

### 9.3 灵饰套装

当 `4 件灵饰` 点化相同特效，且总等级达到 `8 级以上` 时激活。

文档中明确列出的档位有：

- `8 级`
- `16 级`
- `24 级`
- `28 级`
- `32 级`

## 10. 玉魄结构

玉魄至少包括：

- 玉魄名称
- 玉魄品质
- 适用部位
- 契合等级

契合等级范围：

- `1-10 级`

核心属性包括：

- 法术伤害
  - 固定值
  - 百分比
- 法术暴击等级
- 法术忽视 / 穿透
  - 百分比
- 法术伤害结果

文档中明确给出的示例值：

- 固定值示例：`法术伤害 +12`
- 百分比示例：`基础法术伤害 +1.5%`

## 11. 星石结构

星石至少包括四类字段：

- 基础信息字段
- 阴阳状态字段
- 核心属性字段
- 互合规则字段

基础字段包括：

- 星石名称
- 星石类型
- 星石颜色
- 当前状态（阳/阴）
- 星石等级

阴阳状态语义：

- 阳状态：通常增加攻击向属性
- 阴状态：通常增加防御向属性

核心属性示例包括：

- 伤害力
- 法术伤害
- 法术防御
- 气血上限
- 速度
- 物理防御

- 新建同名角色时提示名称已存在

## 14. 本项目的本质

这不是一个纯展示型工具，而是一个：

- 角色数字孪生系统
- 装备识别与审核系统
- 换装推演系统
- 决策辅助系统
- 后台规则配置系统

它最终服务的不是“看面板”，而是回答下面这些问题：

- 这件装备换上后能提升多少法伤
- 技能等级是否会因为符石变化发生跌落
- 总伤提升到底是多少
- 候选装备值不值得买
- 哪套方案最适合当前目标场景

## 15. 后台系统说明

数据库设计需要显式考虑后台，因为本项目大量规则不是一次性写死后就不变的。

后台的核心职责不是“管理用户”，而是“管理规则”。

### 15.1 后台要解决的问题

前台负责：

- 上传截图
- 确认识别
- 入库装备
- 做实验对比
- 查看结果

后台负责：

- 配置成长规则
- 配置技能公式
- 配置符石组合
- 配置星相互合
- 配置灵饰套装档位
- 配置玉魄属性池
- 配置怪物模板
- 配置 OCR 纠错词典
- 发布与回滚规则版本

### 15.2 为什么后台必须纳入一期设计

原因有三个：

- 数值规则本身会变化
- 以后一定会扩门派、扩技能、扩副本
- OCR 识别结果需要持续通过词典和映射修正

如果数据库一期不把后台规则中心考虑进去，后面会出现这些问题：

- 规则被写死在代码里，改一次就要发版
- 无法做规则灰度测试
- 无法回滚历史规则
- 无法审计是谁修改了公式

### 15.3 后台建议拆成三块

1. 规则中心

用于维护：

- 五围成长规则
- 套装规则
- 技能伤害公式
- 星相互合规则
- 玉魄计算规则
- 灵饰套装规则

2. 模板中心

用于维护：

- 怪物模板
- 默认角色模板
- OCR 标准词典
- 装备分类映射

3. 发布中心

用于管理：

- 草稿规则
- 已发布规则
- 回滚版本
- 修改审计日志

### 15.4 后台配置的典型例子

例如当前文档中已经写死的：

- `基础气血来源 * 5 + 体质 * 12 + 耐力 * 4 = 面板气血`
- `1点体质 = 灵力 +0.3`
- `1点魔力 = 魔法上限 +3.5`
- `招云 = 白、红、金、黑、蓝、红`
- `腾蛟 = 白、红、红、黑、蓝、红`
- `龙卷雨击额外伤害 = 目标速度 × 4%`
- `龙腾额外伤害 = 魔法消耗 × 4%`
- `法术忽视 5% = 目标法防按 0.95 参与计算`

这些在产品第一版可以按当前值落地，但数据库和后台能力必须允许后续修改。

### 15.5 后台与前台的关系

建议采用这样的关系：

- 后台维护规则
- 前台不存公式，只读取当前生效版本
- 实验室计算时记录所使用的规则版本

这样做的好处是：

- 结果可追溯
- 历史实验可回放
- 线上问题可快速回滚

## Simulator 默认档案

- 新用户首次进入 simulator 时，如果还没有角色档案，系统会自动创建一份默认角色。
- 默认角色模板优先从 D1 `config` 表读取，当前使用的键包括：
  - `simulator.default.character_meta`
  - `simulator.default.profile`
  - `simulator.default.skills`
  - `simulator.default.cultivations`
  - `simulator.default.equipments`
- 如果配置缺失，服务端会自动补入默认值；后续后台可以直接维护这些配置，而不需要修改查询逻辑。

## Simulator 云端保存

- simulator 当前已经接入 D1 真实读写链路，不再只依赖本地 mock store。
- 读取入口：
  - `GET /api/simulator/current`
- 已完成的写回入口：
  - `PATCH /api/simulator/current/profile`
  - `PATCH /api/simulator/current/cultivation`
  - `PATCH /api/simulator/current/equipment`
- 前端保存入口位于角色面板：
  - 属性页签可保存基础属性与战斗属性
  - 修炼页签可保存修炼等级
  - 装备面板与实验室确认替换可保存当前装备
- 保存成功后，前端会使用接口返回的最新 bundle 重新灌入 store，保证页面状态与 D1 保持一致。

## Simulator 后台模板配置

- 管理后台新增了 simulator 默认模板编辑页：
  - `/admin/simulator-defaults`
- 该页面直接维护 D1 `config` 表中的 `simulator.default.*` 配置。
- 这些模板只影响后续新建档案，不会覆盖已有用户的角色数据。

## 9. 服务端伤害规则链路（新增，2026-04）

- 当前伤害规则已开始迁移到服务端规则中心，规则数据存储在 D1 的 `rule_*` 表中。
- 第一阶段仅覆盖 `龙宫 / 法师 / 龙卷雨击` 主链路。
- 服务端计算默认读取当前生效的 `rule_version`，并组合：
  - `rule_attribute_conversion`
  - `rule_skill_formula`
  - `rule_damage_modifier`
  - `rule_skill_bonus`
- 新增接口 `POST /api/simulator/calculate-damage`，用于根据角色快照、目标参数、战斗上下文和规则版本进行试算。
- 实验室新增接口 `POST /api/simulator/current/lab-valuation`，用于把样本席位 / 对比席位的总伤、差价和性价比统一交给服务端规则中心试算。
- 第一阶段的服务端伤害输出已包含 `breakdown` 明细，便于前端展示公式拆解和规则命中情况。
- 当前试算链路里的 `气血 / 面板法伤 / 魔法值 / 命中 / 伤害 / 物防 / 法防 / 速度 / 躲避` 已统一改为服务端规则推导。
- 实验室页面里的“服务端总伤 / 1点伤害成本”已切换为优先读取服务端试算结果；仅在请求未返回时才短暂回退旧前端展示值。
- 当前生效版本对齐 `gameLogic` 口径：
  - 气血 = `baseHp * 5 + 体质 * 12 + 耐力 * 4 + 装备气血`
  - 面板法伤 = `魔力 * 5 + 灵力 * 1.2 + 等级 * 3 + 装备法伤`
  - 魔法值 = `魔力 * 1.6 + 灵力 * 0.25 + 装备魔法`
  - 命中 = `力量 * 2 + 等级 * 6 + 装备命中`
  - 伤害 = `力量 * 8 + 等级 * 6 + 装备伤害`
  - 物防 = `耐力 * 4 + 体质 * 2 + 等级 * 3 + 装备防御`
  - 法防 = `灵力 * 0.6 + 耐力 * 2 + 等级 * 2.6 + 装备法防`
  - 速度 = `敏捷 * 4 + 等级 * 2 + 装备速度`
  - 躲避 = `敏捷 * 2 + floor(等级 * 0.8) + 装备躲避`
- `rawBodyJson.baseHp` 作为气血规则的基础来源字段持久化；历史快照缺少该字段时，服务端按当前面板气血、体质、耐力和装备气血反推。
- `profile.hp / profile.magicDamage / profile.mp / profile.hit / profile.damage / profile.defense / profile.magicDefense / profile.speed / rawBodyJson.dodge` 不再作为正式试算口径，只保留为角色快照展示字段。
- 后台规则试算页仍允许显式传入“面板法伤覆盖值”，用于调试和回归验证，不影响正式前台试算口径。

## 10. 规则试算样例（新增，2026-04）

- 后台 `Simulator` 页面新增了规则试算样例能力。
- 样例会保存规则版本、试算输入、期望结果快照和备注，持久化到 `rule_simulation_case`。
- 规则编辑后，可以直接载入样例重新试算，用同一组参数比较新旧版本结果。
- 这套机制主要用于伤害规则回归验证，减少分灵、五行、阵法、技能加成等配置调整带来的隐性漂移。
