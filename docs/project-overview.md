# 梦幻实验室项目说明

## 1. 项目定位

梦幻实验室是一个面向《梦幻西游》角色养成与换装决策的数值分析系统。

系统目标不是单纯记录角色面板，而是围绕以下三个核心场景展开：

- 同步玩家当前角色状态
- 识别并管理候选装备
- 在实验室中推演换装后的实际收益

最终输出包括：

- 面板属性变化
- 技能等级变化
- 单次或单秒伤害变化
- 总伤增益百分比
- 性价比判断（含跨服费、1 点法伤成本、低性比提醒）

## 2. 核心页面结构

文档中定义了两大主模块：

- 当前状态
- 实验室

### 2.1 当前状态

目标：构建角色的基准档案，同步游戏现状。

包含三个子模块：

- 个人档案
- 当前装备
- 战斗参数

当前实现补充：

- 登录后会读取 `/api/simulator/current`；若当前用户还没有 simulator 角色，服务端会按默认模板自动补建初始角色与默认战斗参数。
- 当前状态页的资料、修炼、装备、战斗参数都已经接入 D1 真实读写，不再只依赖本地 store。
- 技能伤害面板会联动当前目标模板 / 副本目标参数，当前状态页与实验室不会再各自维护一套脱节的目标值。
- 技能伤害详情弹窗当前已补出“过程分解 + 生效规则与来源”解释层，能直接看到主公式结果、环境修正、罗汉、法伤结果、波动、目标法防结果，以及技能加成 / 套装 / 百分比词条等来源摘要。
- 顶部角色切换器已支持“新建角色弹窗 + 多图导入”：可在建档时一次挂载多张人物面板图和当前装备图，其中人物面板图优先作为新角色的 OCR 基准面板来源，装备图则自动进入该角色的候选装备队列。
- 当前状态页已调整为“左侧角色资料 / 中间当前装备 / 右侧最终面板摘要”的主结构；战斗参数不再默认占据整列，而是收纳到右侧的可展开二级面板里。
- 右侧“最终面板”当前直接展示 OCR 基准面板叠加现有装备、修炼、经脉等增量后的结果，方便先看结论，再决定是否展开战斗参数做试算。
- 当前装备、待确认装备和新品装备库卡片已统一改为“默认展示装备图”，不再直接把 OCR 上传截图当主图；若该装备带有 OCR 原图，鼠标悬停图片可直接查看“装备图 vs 上传截图”对比。
- 当前装备页里的“从装备库替换”弹窗，也已切到和实验室一致的统一总库来源：会合并当前方案、其他方案和候选装备库，并在选择卡片上直接标注来源标签，减少在两个入口之间来回找装备时的割裂感。
- 当前装备区已新增独立“装备总库”入口，可直接在前台查看统一库存，并按来源、主类、部位和总价排序筛选，不再必须切到实验室或后台台账才能核对库存构成。
- 当前装备区的“装备总库”已进一步支持快捷挂载：主装备可直接一键挂到当前栏位；灵饰 / 玉魄若缺少明确槽位则会保守退回到“查看详情”，避免误替换。
- 当前装备区的“装备总库”也已接通实验室入口：可直接把一件装备送到实验室对比席位，并自动切到实验室页，减少“看总库 -> 切实验室 -> 再选一遍”的重复操作。
- 当前装备区的“装备总库”现已支持“挂到指定方案”：不切换当前页面，也能把一件装备直接写入某个装备方案，开始真正服务多方案管理，而不只是替换当前穿戴。
- 当前装备区的“装备总库”已开始直接展示方案占用语义：卡片上会补充“方案占用”提示，能更快判断这件装备目前写入了哪些方案，而不只是看到来源标签。
- 当前装备区的“装备总库”现已支持候选装备库直清：已确认候选装备不必再切回实验室或后台，也能直接在总库卡片上执行“移出候选库”，并同步回落库存待用计数与未写入方案统计。
- 当前装备区的“装备总库”现已进一步支持候选库批量清理：切到“候选装备库”来源后，可按当前筛选结果一键批量移出候选来源，适合清掉一轮已判定无用的候选。
- 当前状态页在刷新或重新进入后，也会直接同步候选装备库，不再依赖“先进入实验室”才能把候选库存带进总库视图；当前页提示文案也已同步更新。
- 当前装备区的“挂到指定方案”弹层也已补覆盖提示：现在会明确区分“该方案已存在同一件装备 / 将替换当前同部位装备 / 该栏位为空”，减少误覆盖。
- 当前装备区的“装备总库”现已支持按方案直接筛库存：可以切到某一套方案，或查看“未写入方案”的库存视角，让总库开始具备按方案盘货的能力。
- 当前装备区的“装备总库”现已补上方案反向清理：切到某套装备方案后，单张卡片可直接“移出该方案”，同时也支持按当前筛选结果批量移出该方案；若目标正是当前方案，还会同步更新当前穿戴。
- 当前装备区的“装备总库”在方案移出后也会立即刷新当前筛选结果；像“高速方案”这类方案视角下，被移出的装备会当场从列表消失，不再残留旧的来源标签。
- 当前装备区的“装备总库”现已识别实验室样本方案：会直接展示“当前方案 / 实验室样本方案”的关系，并支持一键切到这两个方案视角，减少当前页与实验室之间来回对照的理解成本。
- 当前装备区的“装备总库”现已补当前方案与实验室样本方案的差异统计：会直接显示差异部位数量，并支持一键只看这些差异部位，帮助先锁定值得进实验室细看的栏位。
- 当前装备区的“装备总库”现已把差异部位继续细分成三类：`当前方案独有 / 实验室样本独有 / 同部位不同装备`，可以直接按差异性质筛选，更快判断是补位、删位还是直接替换。
- 当前装备区的“装备总库”现已补出几组快捷处理视角，例如 `待送实验室差异`、`当前方案独有且未进实验室`、`同部位替换候选`，减少手动叠加筛选条件的次数。
- 这些快捷处理视角当前已收口为独立推荐视角，不再简单依赖“库存待用”状态叠加；像“当前正在穿戴、但还没送进实验室”的差异装备，现在也能被正常筛出来。
- 当前装备区的“装备总库”卡片已开始跟随快捷处理视角给出“建议动作”，并高亮对应按钮，例如优先提示“送去实验室”或“挂到当前”，减少筛出来之后还要自己二次判断下一步操作。
- 在 `待送实验室差异 / 当前方案独有且未进实验室` 这两类视角下，总库当前还会补“批量送去实验室”入口；批量动作会按当前排序对部位去重，同部位只取最靠前的一件，避免候选互相覆盖。
- 在 `同部位替换候选` 视角下，总库当前也已补“批量写入方案”入口：会先按当前排序对部位去重，再弹出方案选择层，并明确显示每套方案会有多少件是 `直接写入 / 覆盖替换 / 原本已一致`。
- 批量写入方案当前已支持三种模式：`全部可变动项`、`仅补空栏位`、`仅替换已有栏位`。如果只是想往某套方案里补缺口，或只想替换已有部位、不往空栏位新增装备，都已经可以直接在弹层里切换。
- 当前装备区的总库在执行批量动作后，顶部还会保留一条“结果摘要”，直接展示这次影响到的目标去向和部位列表；同时会自动切到更合适的筛选视角，例如送实验室后切到 `实验室对比中`、批量写方案后切到目标方案。
- 这条结果摘要当前也不只是展示信息，已经补了两个直接动作：`只看本次部位` 与 `回到目标视图`。也就是说，批量处理完之后可以直接把总库收窄到本次受影响部位，或者一键回到系统推荐的核对视角。
- 结果摘要条现在还可以直接继续下一步处理：如果本次部位里还有可送实验室、可移出实验室、或可继续写入方案的装备，会直接在摘要条里给出对应批量动作入口，不必重新找回原来的操作面板。
- 结果摘要条里当前还补了“本次部位处理进度”统计，会直接显示这批部位里当前有多少还在穿戴、多少在库存待用、多少已在实验室、以及还能继续做哪些批量动作，便于判断这一轮处理是否已经收尾。
- 这块进度统计现在也已经进一步收成“阶段状态”，例如 `待送实验室 / 实验室处理中 / 待写入方案 / 已收尾`，并附带简单的收尾进度条，方便一眼判断这批部位是否还需要继续处理。
- 阶段状态现在还会直接驱动“推荐下一步”动作。也就是说，当系统判断这批部位当前最适合继续送实验室、撤回实验室，或写入方案时，会把对应按钮自动放到更靠前的位置并高亮展示。
- 当系统判断这批部位已经 `已收尾` 时，结果摘要条现在也会切换成更明确的完成态：不再继续强调后续动作，而是直接提示“可以完成本轮”，并提供收尾入口。
- 在 `已收尾` 的完成态下，结果摘要条现在还补了“开始下一批”入口；会直接退出本轮摘要聚焦，并重置快捷视角相关筛选，方便继续处理后续库存，而不是停留在上一批的上下文里。
- 当前装备区的“装备总库”还补了状态可视化：卡片会直接标记这件装备是“当前穿戴”“已在实验室对比席位”还是“库存待用”，并在已送入实验室时禁用重复投递动作。
- 当前装备区的“装备总库”已支持实验室回流：已在对比席位里的装备会切换为“移出实验室”动作，可直接从总库撤回，不用再切到实验室逐槽移除。
- 当顶部切到 `实验室对比中` 状态视图时，总库当前也已补“批量移出实验室”入口，方便一次清掉当前筛选结果里已经比完的一批装备。
- 当前装备区的“装备总库”顶部汇总卡片已改成可点击筛选：现在可以直接按“当前穿戴 / 实验室对比中 / 库存待用”切换结果，不用再靠肉眼扫卡片状态。
- 装备 OCR 上传入口当前已补“截图类型提示”，默认使用 `自动识别`，也支持手动切换 `通用装备图 / 藏宝阁截图 / 聊天框预览`；服务端会按该提示或自动判断结果调整识别关注重点。
- OCR 审计链当前也会把本次使用的截图提示模式写入 raw 结果元信息，后台排障时可以区分“自动识别”还是“手动指定”触发的识别结果。
- 当前状态页的 OCR 识别记录，以及后台 OCR 作业详情页，也已经直接展示这层截图提示模式，排查识别偏差时不需要再翻 raw JSON。
- 刚识别完成的装备确认弹窗，以及实验室待确认详情，也会同步展示这层截图提示模式，人工审核时能直接知道这次是按哪种截图口径识别的。
- 当前前台装备图已不再依赖外部随机图；现统一改为本地 `equipment-art` resolver，优先按装备名命中素材映射，命不中时回退到按部位生成的本地占位图。
- 真实梦幻装备素材当前已接入本地 `素材/icons` 首批图片，并已批量导入到 `public/simulator/equipment-art/` 与 artwork manifest；前台会优先命中本地真实装备图，命不中时再回退到部位占位图。
- 当前装备页已在玉魄区下方补出独立“神器加成”编辑区，可手动选择单一属性、填写数值与说明，并单独同步到云端；刷新后会跟随角色档案一起恢复。
- 神器加成编辑区当前已补入常用预设模板和亮点标签展示，便于快速切换法伤 / 法结 / 法暴 / 速度 / 穿刺等实验场景。

### 2.2 实验室

目标：做换装推演与购买决策。

包含三个子模块：

- 装备库
- 实验对比中心
- 预期效果预判

当前实现里，实验室左侧的“待确认新品 / 新品装备库”已经接入 D1 持久化，使用独立的候选装备表保存候选库存。
当前实现里，右侧 AI 顾问面板已经接入 Gemini，会结合当前角色、候选装备和实验室席位上下文回答配装问题。
当前实现里，实验室支持把单件或整套实验结果直接应用到当前装备；应用前会自动生成 `应用前快照`，并允许回滚最近一次应用。
当前实现里，OCR 已接入 `Gemini + R2`，后台已提供 OCR 健康检查、OCR 作业、OCR 字典、候选装备库与入库台账页面。
当前实现里，前台 OCR 已统一先写入 `ocr_job -> ocr_draft_item` 作为审计链，再自动同步到 `candidate_equipment.pending`；当前不再依赖后台逐条人工处理才能进入候选装备库。
当前实现里，后台治理页已继续补齐到第二阶段：新增了装备扩展规则工作台、OCR 质量统计页和顾问问答审计页。
当前后台已覆盖导航、规则中心、星相互合规则、默认模板、目标模板、OCR 运营页、候选装备库、入库台账、实验室记录、用户排障与顾问配置。
当前实验室正式口径为“样本席位 + 1 个对比席位”，并已补齐继承模拟、伤害差额显示、横向对比和替换确认主链。
当前实验室待确认区已支持批量上传装备截图，会按顺序逐张 OCR 并给出批量完成提示。
当前实验室左侧装备库已进一步收口为“双层视图”：待确认区仍按 OCR 候选审核流工作，第二个 tab 则升级为“装备总库视图”，会合并当前方案、其他装备方案和已确认候选装备；总库默认仍支持按主类 / 部位筛选，并保留总价排序。
当前实验室左侧的“装备总库视图”也已补来源筛选条，可直接切到 `全部来源 / 当前方案 / 其他方案 / 候选装备库` 四种来源视角。
当前实验室左侧的“装备总库视图”也已补候选来源清理动作：若一件装备同时来自候选库和某个方案，执行“移出候选库”只会去掉候选来源，不会误删方案内装备。
当前实验室总库详情弹窗也已补“来源归属”区，可直接看到装备来自哪些方案/库存上下文；如果该装备带有候选来源，也能直接在详情里发起“移出候选库”。
当前实验室总库卡片也已直接展示“方案占用摘要”：若某件装备已写入多个方案，会显示 `方案占用：当前方案 / 高速方案 等 N 套`；如果只是候选装备库里的独立候选，则会显示 `当前未写入任何装备方案`。若候选装备字段还不完整，卡片会退化显示 `待补充属性`，不会因为缺少 `mainStat` 把整块总库视图打崩。
当前实验室总库卡片也已支持直接“挂到对比席位 / 移出对比位”：样本位本来就继承的装备会显示 `样本已继承`，避免把当前穿戴误当成实验差异；真正的候选差异装备则可直接从卡片送进 `对比席位1`，挂入后再原位撤回，不必先开详情弹窗。
当前前台两处总库入口也已开始并入“正式库存”来源：系统会从当前角色的有效 `inventory_entry / inventory_equipment_asset` 读取正式库存资产，并和当前方案、其他方案、候选装备一起合并展示；来源筛选也已新增 `正式库存` 视角，开始把前台总库和后台正式库存台账往同一套语义上收。
当前实验室总库也已补上正式库存状态动作：若一件装备来自候选确认后生成的正式库存，卡片和详情弹窗都可直接执行 `标记已售出 / 标记作废`；这类动作会同步回写关联候选来源，但不会影响仍在实验席位或装备方案中的引用。

## 14. 本项目的本质

这不是一个纯展示型工具，而是一个：

- 角色数字孪生系统
- 装备识别与审计系统
- 换装推演系统
- 决策辅助系统
- 后台规则配置系统

它最终服务的不是“看面板”，而是回答下面这些问题：

- 这件装备换上后能提升多少法伤
- 技能等级是否会因为符石变化发生跌落
- 总伤提升到底是多少
- 候选装备值不值得买
- 哪套方案最适合当前目标场景

## 15. 后台系统说明

数据库设计需要显式考虑后台，因为本项目大量规则不是一次性写死后就不变的。

后台的核心职责不是“管理用户”，而是“管理规则”。

### 15.1 后台要解决的问题

前台负责：

- 上传截图
- 确认识别
- 入库装备
- 做实验对比
- 查看结果

后台负责：

- 配置成长规则
- 配置技能公式
- 配置符石组合
- 配置星相互合
- 配置灵饰套装档位
- 配置玉魄属性池
- 配置怪物模板
- 配置 OCR 纠错词典
- 发布与回滚规则版本

### 15.2 为什么后台必须纳入一期设计

原因有三个：

- 数值规则本身会变化
- 以后一定会扩门派、扩技能、扩副本
- OCR 识别结果需要持续通过词典和映射修正

如果数据库一期不把后台规则中心考虑进去，后面会出现这些问题：

- 规则被写死在代码里，改一次就要发版
- 无法做规则灰度测试
- 无法回滚历史规则
- 无法审计是谁修改了公式

### 15.3 后台建议拆成三块

1. 规则中心

用于维护：

- 五围成长规则
- 套装规则
- 技能伤害公式
- 星相互合规则
- 玉魄计算规则
- 灵饰套装规则

2. 模板中心

用于维护：

- 怪物模板
- 默认角色模板
- OCR 标准词典
- 装备分类映射

3. 发布中心

用于管理：

- 草稿规则
- 已发布规则
- 回滚版本
- 修改审计日志

当前实现补充：

- 已落地的后台页包括：默认模板、规则中心、规则 playground、装备扩展规则、星相互合规则、目标模板、OCR 健康检查、OCR 作业、OCR 质量统计、OCR 字典、待确认 / 审计台、候选装备库、入库台账、顾问配置、顾问问答审计、实验室记录、用户排障。
- 后台能力目前已经覆盖“规则维护 + OCR 运营 + 用户排障 + 顾问审计”的主链；剩余缺口主要在更细的 OCR 强校验和更丰富的质量指标，而不是后台入口缺失。

### 15.4 后台配置的典型例子

例如当前文档中已经写死的：

- `基础气血来源 * 5 + 体质 * 4.5 = 面板气血`
- `1点体质 = 灵力 +0.3`
- `1点魔力 = 魔法上限 +3.5`
- `招云 = 白、红、金、黑、蓝、红`
- `腾蛟 = 白、红、红、黑、蓝、红`
- `龙卷雨击额外伤害 = 目标速度 × 4%`
- `龙腾额外伤害 = 魔法消耗 × 4%`
- `法术忽视 5% = 目标法防按 0.95 参与计算`

这些在产品第一版可以按当前值落地，但数据库和后台能力必须允许后续修改。

### 15.5 后台与前台的关系

建议采用这样的关系：

- 后台维护规则
- 前台不存公式，只读取当前生效版本
- 实验室计算时记录所使用的规则版本

这样做的好处是：

- 结果可追溯
- 历史实验可回放
- 线上问题可快速回滚

## 16. 运行时与持久化说明

### 16.1 默认档案

- 新用户首次进入 simulator 时，如果还没有角色档案，系统会自动创建一份默认角色。
- 默认角色模板优先从 D1 `config` 表读取，当前使用的键包括：
  - `simulator.default.character_meta`
  - `simulator.default.profile`
  - `simulator.default.skills`
  - `simulator.default.cultivations`
  - `simulator.default.equipments`
- 如果配置缺失，服务端会自动补入默认值；后续后台可以直接维护这些配置，而不需要修改查询逻辑。

### 16.2 云端保存

- simulator 当前已经接入 D1 真实读写链路，不再只依赖本地 mock store。
- 读取入口：
  - `GET /api/simulator/current`
- 已完成的写回入口：
  - `PATCH /api/simulator/current/profile`
  - `PATCH /api/simulator/current/cultivation`
  - `PATCH /api/simulator/current/equipment`
  - `GET /api/simulator/current/equipment/rollback`
  - `POST /api/simulator/current/equipment/rollback`
- 前端保存入口位于角色面板：
  - 属性页签可保存基础属性与战斗属性
  - 修炼页签可保存修炼等级
  - 装备面板与实验室确认替换可保存当前装备
- 战斗参数保存现在除了当前单目标上下文，还会一并写回手动目标列表、副本勾选状态和当前 tab。
- 实验室里的“单件同步到当前装备 / 整套应用到当前装备”现在会先自动创建一份 `应用前快照`，再覆写当前装备。
- 实验室头部新增“回滚最近一次应用”入口，会把最近一次自动保存的应用前快照重新写回当前 `current` 快照，而不是直接切换 `current_snapshot_id`，避免历史快照后续被继续编辑污染。
- 保存成功后，前端会使用接口返回的最新 bundle 重新灌入 store，保证页面状态与 D1 保持一致。

### 16.3 后台模板配置

- 管理后台新增了 simulator 默认模板编辑页：
  - `/admin/simulator-defaults`
- 该页面直接维护 D1 `config` 表中的 `simulator.default.*` 配置。
- 这些模板只影响后续新建档案，不会覆盖已有用户的角色数据。
- 目标模板后台页现在同时维护两类目标：
  - 副本目标模板
  - 手动目标模板
- 手动目标模板的扩展攻击/抗性字段保存在 `battle_target_template.payload_json`，首页手动目标面板会直接读取。
- 系统会预置一组副本目标模板，当前默认包含 `乌鸡国 / 水陆大会 / 车迟国 / 大雁塔 / 通天河 / 平顶山 / 红孩儿 / 齐天大圣 / 石猴授徒` 九组 fallback 样例副本；后台模板存在时优先读后台，缺失时自动补齐前端内置模板。

## 17. 服务端伤害规则链路

当前服务端试算已经成为正式规则入口，前台技能伤害面板、实验室估值和后台规则试算共用同一套链路。

### 17.1 当前已落地能力

- 规则中心已接管 `rule_attribute`、`rule_skill_formula`、`rule_damage_modifier`、`rule_skill_bonus` 等核心规则表，服务端默认按当前生效 `rule_version` 试算。
- 当前已覆盖 `龙宫 / 法师` 的 `龙卷雨击 / 龙腾` 主链路，服务端会统一推导 `气血 / 法伤 / 法防 / 速度 / 命中 / 伤害 / 物防 / 躲避` 等正式面板。
- `POST /api/simulator/calculate-damage` 负责角色伤害试算，`POST /api/simulator/current/lab-valuation` 负责实验室总伤、差价和性价比估值。
- 技能试算与实验室估值都会继承同一份战斗上下文，包括阵法、五行、目标法防、目标速度、修炼差和附加结果项，不再各自维护一套条件。
- 服务端返回 `breakdown` 明细，前台可直接展示公式拆解与规则命中情况。

### 17.2 装备扩展规则

- 当前装备与实验室席位会把结构化宝石、符石组合、星位和星相互合配置一并持久化，服务端按这些结构化数据自动命中规则。
- 灵饰与玉魄已经独立建模，当前快照、历史回滚、默认角色补建和实验室估值都走同一套持久化链路。
- 灵饰套装规则已进入运行时，当前按“4 件同名灵饰 + 总等级档位”识别套装效果。
- 常规套装规则已进入运行时，当前优先读取 `regular_set_rules`；若后台未配置，则回退到临时口径 `3 件 +10 魔力 / 5 件 +20 魔力`。
- 玉魄百分比词条已进入正式计算，当前覆盖 `法术忽视 / 穿透 %`、`基础法术伤害 %`、`魔法值上限 %`、`五行属性增强 %`。
- 玉魄属性池和百分比语义已经接入规则中心，并开始约束前台编辑器和 OCR 确认弹窗。

### 17.3 符石、星相与战斗规则

- `招云 / 腾蛟` 两套 6 件符石套装已经接入正式伤害修正，会追加灵力收益和技能结果伤害。
- `九龙诀 / 呼风唤雨 / 破浪诀 / 逆鳞 / 龙腾` 按全身只取最高一套处理，`隔山打牛` 按最多两套生效处理。
- 当前系统已经按正式口径处理符石套装失效、额外灵力拆分、技能等级差值联动和基线收益去重。
- 星相互合与 6 件全套 +2 当前优先读取结构化配置；由于 `招云` 武器首孔颜色与 `破浪诀` 颜色清单存在规则冲突，现阶段仍保留一层折中校验口径，后续规则确认后可继续收紧。
- 实验室已支持符石技能变化、跌落预警、孔数冲突提示和整套应用前确认框。
- 实验室目标选择器与技能伤害面板都会按最终技能等级动态刷新“秒几”上限。

### 17.4 当前统一口径

## 18. 规则试算样例

- 后台 `Simulator` 页面新增了规则试算样例能力。
- 样例会保存规则版本、试算输入、期望结果快照和备注，持久化到 `rule_simulation_case`。
- 规则编辑后，可以直接载入样例重新试算，用同一组参数比较新旧版本结果。
- 这套机制主要用于伤害规则核对，减少分灵、五行、阵法、技能加成等配置调整带来的隐性漂移。
